On Tuesday, July 29, 2014, following planned datacenter maintenance, Yeller experienced multiple network partitions within its server cluster starting around 12:00 UTC. This led to severe delays in exception processing and disabled write operations for user account and billing data. The issue persisted for approximately 21.5 hours.

Customers experienced two primary impacts. Firstly, 42 exceptions saw processing delays of up to 7 hours, requiring manual reprocessing due to Riak write timeouts. Secondly, users were unable to perform actions requiring writes to the user/billing data store, such as signing up, changing passwords, or inviting users. While these functions were impaired, Yeller remained available for reading exceptions, and no exceptions were lost.

The network partitions were eventually resolved on Wednesday, July 30, 2014, around 09:30 UTC, after a careful rolling restart of the cluster. The exact root cause of the partition remains unclear, though cached routes are suspected. The restart, while effective, unfortunately cleared the machine state that could have aided in definitive root cause analysis.

In response to the incident, Yeller plans to improve outage communication by utilizing Statuspage.io and internal chat bots, and by displaying system status in the web UI. Technical improvements include investigating enhanced retry logic for exception writes to reduce manual intervention. Future incident response will aim to preserve the state of affected machines for better root cause determination.
