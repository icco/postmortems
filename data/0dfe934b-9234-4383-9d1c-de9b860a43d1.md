In late February 2017, the WebKit Subversion repository experienced an incident after a user committed two files that demonstrated a recently discovered SHA1 hash collision. These files, despite having different content, produced the same SHA1 hash, leading to unexpected behavior within the repository.

The root cause was Subversion's "Representation Sharing" feature, introduced in version 1.6. This feature uses SHA1 hashes to deduplicate file content, storing only a pointer if the same hash is encountered. When the two colliding files were committed, the second file was effectively replaced by a pointer to the first, leading to data corruption.

Customer impact manifested as "Checksum mismatch" errors when users attempted to checkout or update the problematic file. This rendered the affected file inaccessible and also caused issues for tools like `svnsync` and `git-svn` that replay transaction histories, as they would fail upon encountering the corrupted revision.

Remediation options discussed include disabling the representation sharing feature, implementing pre-commit hooks to block known colliding files, or deleting the problematic file from the HEAD revision. The WebKit repository specifically addressed the issue by creating a Subversion permission rule (authz) to block access to the affected files, which allowed for a more robust long-term solution including potential migration to a new repository.
