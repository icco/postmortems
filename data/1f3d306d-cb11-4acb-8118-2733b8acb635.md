On November 2, 2020, Cloudflare experienced an incident that impacted the availability of its API and dashboard for six hours and 33 minutes. The issue began at 14:43 UTC with a partial network switch failure. This misbehaving switch, while not fully down, caused inconsistent network traffic, leading to etcd cluster errors and repeated leader elections.

The partial switch failure created a "Byzantine fault" scenario for the etcd cluster, where nodes received conflicting information, preventing a stable leader election. This made the etcd cluster read-only. Subsequently, a defect in the database cluster management system, which relied on etcd, triggered the promotion of a new primary database but required a full rebuild of all replicas, causing significant delays.

During the incident, the success rate for API queries periodically dropped to 75%, and the dashboard experience was up to 80 times slower than normal. The primary authentication database became overloaded due to the lack of available replicas, impacting API calls and dashboard activities.

Cloudflare manually reduced load by dialing back non-critical services and diverted API read-queries to replicas in a secondary data center to improve availability. The incident concluded at 21:20 UTC when the first database replica rebuilt and performance returned to normal.

Post-incident, Cloudflare adjusted the configuration for automatic database replica promotion to prevent overly quick triggers. They also fixed the bug in the cluster management system that necessitated full replica rebuilds and are redesigning the user session system for better cross-data center flexibility.
