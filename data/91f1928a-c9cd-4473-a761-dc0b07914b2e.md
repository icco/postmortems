On February 18, 2017, Google's Project Zero reported a security vulnerability in Cloudflare's edge servers. A bug in Cloudflare's HTML parser caused servers to return memory containing private information such as HTTP cookies, authentication tokens, and HTTP POST bodies. The issue had been dormant for years but became active with the rollout of certain features, with the earliest potential leak date being September 22, 2016.

The root cause was a buffer overrun in Cloudflare's Ragel-based HTML parser. A pointer error, specifically a missing `fhold` in an error handler, allowed the parser to read past the end of a buffer. This latent bug was exposed when the introduction of a new parser, `cf-html`, subtly changed the internal buffer handling within NGINX, causing `last_buf` to be set to 1 in specific scenarios, triggering the error path.

The leaked memory could contain sensitive customer data, including HTTP headers, POST data, JSON for API calls, URI parameters, cookies, API keys, and OAuth tokens. While Cloudflare customer SSL private keys were not affected, an internal private key used for machine-to-machine encryption was leaked. The greatest period of impact was from February 13 to February 18, 2017, affecting approximately 1 in every 3.3 million HTTP requests.

Cloudflare quickly mitigated the issue by disabling three features (email obfuscation, Server-side Excludes, and Automatic HTTPS Rewrites) that used the vulnerable parser chain. The initial mitigation took 47 minutes from receiving the report, with global resolution achieved within 7 hours. Cloudflare also worked with Google, Yahoo, and Bing to identify and purge 770 unique URIs containing leaked memory from search engine caches before public disclosure.

The incident highlighted that a latent bug in older software could be exposed by new system interactions. Cloudflare's infosec team initiated a project to fuzz older software for other potential security problems and added explicit pointer checks to generated code to prevent future buffer overruns.
