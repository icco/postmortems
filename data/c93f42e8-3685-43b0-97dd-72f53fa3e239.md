A bug in the Linux kernel's timekeeping mechanism, specifically concerning CLOCK_REALTIME and hrtimer handling during a leap second event, led to widespread system instability and high CPU load in 2012. The issue was identified and discussed in July 2012, with a proposed fix.

The core problem occurred when a leap second was applied, causing CLOCK_REALTIME to be rewound by one second. Crucially, the clock_was_set() function was not called during this process, which meant the hrtimer base.offset for CLOCK_REALTIME was not updated. Consequently, the hrtimer code's internal sense of wall time became one second ahead of the system's actual timekeeping core.

This discrepancy caused TIMER_ABSTIME CLOCK_REALTIME timers to expire one second prematurely. More severely, sub-second TIMER_ABSTIME CLOCK_REALTIME timers would return immediately. Applications that relied on these short timers in a loop, such as those using futex_wait or similar timeout mechanisms, would enter a spinwait state, consuming excessive CPU resources and leading to significant load spikes across affected systems. This behavior was observed to cause numerous web services to experience outages.

The proposed solution involved modifying the kernel to ensure that clock_was_set() is explicitly called when a leap second is applied. This change would correctly synchronize hrtimer base.offset with the updated CLOCK_REALTIME, thereby preventing timers from expiring prematurely and resolving the high load issue. The author confirmed that with this fix, the described behavior no longer occurred.
